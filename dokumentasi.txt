1. ALUR SISTEM
========================================================================

----------------------------------------------------------------------
1.1. Alur Registrasi Mahasiswa
----------------------------------------------------------------------
   Mulai
     |
     v
   [Super Admin / Admin Akademik]
     |
     +---> Input Data Mahasiswa (manual)
     |       - NIM
     |       - Nama Lengkap
     |       - Tempat, Tanggal Lahir
     |       - Jenis Kelamin
     |       - Program Studi (Prodi)
     |       - Tahun Masuk
     |       - Email, No HP
     |       - Status (aktif/lulus/dropout)
     |
     +---> Import Data Massal (file Excel/CSV)
     |
     v
   Data tersimpan di database MySQL
     |
     v
   Akun login dibuat untuk mahasiswa (opsional)
     |
     v
   [Selesai]


----------------------------------------------------------------------
1.2. Alur Penerbitan Ijazah
----------------------------------------------------------------------
   Mulai
     |
     v
   [Admin Akademik] pilih menu "Sertifikat"
     |
     v
   Pilih mahasiswa yang sudah lulus (status = 'lulus')
     |
     v
   Sistem generate:
     - Nomor Ijazah: UNSUB/{kode_prodi}/{tahun}/{random_6_char}
     - SHA-256 Hash: hash('sha256', {nim + nama + prodi + ipk + nomor_ijazah + tanggal_lulus + timestamp})
     - QR Code: SVG yang berisi URL verifikasi (frontend/verify/{hash})
     |
     v
   Status ijazah: "draft"
     |
     v
   [Selesai]


----------------------------------------------------------------------
1.3. Alur Publikasi ke Blockchain
----------------------------------------------------------------------
   Mulai
     |
     v
   [Admin Akademik] pilih ijazah status "draft"
     |
     v
   Klik tombol "Publish to Blockchain"
     |
     v
   Backend memanggil BlockchainService::issueCertificate()
     |
     +---> [Jika Mock Mode = true]
     |       Generate transaction hash palsu (0x + 32 byte random hex)
     |       Simpan ke tabel blockchain_transactions (status = 'confirmed')
     |
     +---> [Jika Mock Mode = false / Production]
     |       Kirim transaksi ke smart contract Ethereum via JSON-RPC
     |       Tunggu konfirmasi dari jaringan
     |
     v
   Update data ijazah:
     - blockchain_tx_hash = transaction hash
     - blockchain_block = nomor block
     - blockchain_timestamp = waktu konfirmasi
     - status = 'issued'
     - issued_at = now()
     |
     v
   Generate PDF Ijazah (DomPDF):
     - Template ijazah resmi
     - Data mahasiswa lengkap
     - QR Code (base64 embedded)
     - Link verifikasi
     - Link Etherscan区块链
     |
     v
   PDF disimpan di storage
   Hash PDF dihitung dan disimpan (pdf_hash)
     |
     v
   [Selesai]


----------------------------------------------------------------------
1.4. Alur Verifikasi Ijazah (Publik)
----------------------------------------------------------------------
   Mulai
     |
     v
   [Visitor / Publik] membuka halaman verifikasi
     |
     +---> Metode 1: Verifikasi via Hash
     |       Masukkan SHA-256 hash (64 karakter hex)
     |           |
     |           v
     |       POST /api/verify/hash
     |           |
     |           v
     |       Cari di tabel ijazah berdasarkan hash_sha256
     |       Verifikasi ke blockchain via BlockchainService::verifyCertificate()
     |           |
     |           v
     |       Log verifikasi dicatat (IP, user agent, timestamp, hasil)
     |           |
     |           v
     |       Tampilkan hasil: VALID / INVALID
     |           - Jika valid: nama, NIM, prodi, IPK, fakultas, dll
     |           - Jika revoked: status dicabut
     |
     +---> Metode 2: Verifikasi via Upload File PDF
     |       Upload file PDF (max 5MB)
     |           |
     |           v
     |       Sistem hitung SHA-256 hash dari file PDF
     |       Cari di tabel ijazah berdasarkan pdf_hash
     |       Verifikasi ke blockchain via BlockchainService::verifyCertificate()
     |           |
     |           v
     |       Tampilkan hasil (sama seperti metode hash)
     |
     +---> Metode 3: Verifikasi via QR Code
     |       Scan QR Code dari PDF ijazah
     |           |
     |           v
     |       QR berisi URL: frontend/verify/{hash}
     |           |
     |           v
     |       Langsung menuju halaman verifikasi dengan hash
     |       Proses sama seperti verifikasi via hash
     |
     v
   [Selesai]


----------------------------------------------------------------------
1.5. Alur Revokasi Ijazah
----------------------------------------------------------------------
   Mulai
     |
     v
   [Admin Akademik] pilih ijazah status "issued"
     |
     v
   Klik "Revoke Certificate"
     |
     v
   Update status ijazah menjadi "revoked"
   Catat notes alasan revokasi
     |
     v
   Verifikasi selanjutnya akan menampilkan "REVOKED"
     |
     v
   [Selesai]

3. SPESIFIKASI KEBUTUHAN SISTEM (SRS)
========================================================================

----------------------------------------------------------------------
3.1. Tujuan Sistem
----------------------------------------------------------------------
  Sistem Ijazah Digital Berbasis Blockchain Universitas Subang bertujuan
  untuk:
    1. Mendigitalisasi proses penerbitan ijazah
    2. Mengamankan data ijazah menggunakan teknologi blockchain
    3. Menyediakan verifikasi ijazah secara publik dan instan
    4. Mencegah pemalsuan ijazah
    5. Menyediakan audit trail yang lengkap

----------------------------------------------------------------------
3.2. Fitur-Fitur Utama (Functional Requirements)
----------------------------------------------------------------------
  FR-01: Autentikasi Pengguna
    - Login dengan email dan password
    - Logout
    - Ganti password
    - Token-based authentication (Sanctum)

  FR-02: Manajemen Mahasiswa
    - CRUD data mahasiswa
    - Import mahasiswa massal
    - Filter berdasarkan fakultas/prodi
    - Search berdasarkan NIM/nama

  FR-03: Manajemen Ijazah/Sertifikat
    - Generate ijazah (draft)
    - Update metadata ijazah
    - Publish ke blockchain
    - Download PDF ijazah
    - Revoke ijazah
    - Status workflow: draft -> pending -> issued -> revoked

  FR-04: Verifikasi Ijazah (Publik)
    - Verifikasi via SHA-256 hash
    - Verifikasi via upload file PDF
    - Verifikasi via QR Code (redirect URL)
    - Tampilkan detail mahasiswa jika valid
    - Tampilkan status revoked jika dicabut

  FR-05: Dashboard & Statistik
    - Statistik real-time (total alumni, ijazah terbit, verifikasi)
    - Grafik dan chart (Recharts)
    - Role-based dashboard view

  FR-06: Role-Based Access Control
    - Super Admin: akses penuh, manajemen user
    - Admin Akademik: manajemen mahasiswa & ijazah
    - Verifikator: lihat log verifikasi
    - Mahasiswa: lihat profil & status ijazah sendiri

  FR-07: Audit Trail
    - Log semua aktivitas pengguna
    - Log semua verifikasi (IP, user agent, hasil)
    - Log transaksi blockchain

----------------------------------------------------------------------
3.3. Kebutuhan Non-Fungsional (Non-Functional Requirements)
----------------------------------------------------------------------
  NFR-01: Keamanan
    - Password di-hash menggunakan bcrypt
    - API dilindungi token (Sanctum)
    - Data di blockchain immutable

  NFR-02: Kinerja
    - Verifikasi ijazah < 3 detik
    - Generate PDF < 10 detik
    - Halaman dashboard load < 3 detik

  NFR-03: Availability
    - Sistem tersedia 24/7
    - Downtime maintenance < 2 jam/bulan

  NFR-04: Usability
    - UI responsif (mobile & desktop)
    - Bahasa Indonesia
    - Dark theme modern

  NFR-05: Scalability
    - Mendukung ribuan alumni
    - Database dapat di-scale vertikal/horizontal

----------------------------------------------------------------------
5.2. Skenario Use Case
----------------------------------------------------------------------

  --- UC-01: Login ---
  Aktor: Super Admin, Admin Akademik, Verifikator, Mahasiswa
  Pre-condition: Pengguna memiliki akun terdaftar
  Trigger: Pengguna membuka halaman login
  Skenario Normal:
    1. Sistem menampilkan form login (email & password)
    2. Pengguna memasukkan email dan password
    3. Sistem memvalidasi kredensial
    4. Sistem membuat token Sanctum
    5. Sistem mengembalikan token dan data user
    6. Sistem redirect ke halaman dashboard sesuai role
  Post-condition: Pengguna berhasil login dan mendapat token akses
  Skenario Alternatif (gagal):
    3a. Email tidak ditemukan -> tampilkan error "Email tidak terdaftar"
    3b. Password salah -> tampilkan error "Password salah"
    3c. Akun nonaktif -> tampilkan error "Akun telah dinonaktifkan"


  --- UC-02: Generate Ijazah ---
  Aktor: Admin Akademik
  Pre-condition: Admin sudah login dan ada mahasiswa dengan status 'lulus'
  Trigger: Admin memilih mahasiswa lalu klik "Generate Ijazah"
  Skenario Normal:
    1. Admin pilih mahasiswa dari daftar
    2. Sistem menampilkan form input IPK dan tanggal lulus
    3. Admin memasukkan IPK dan tanggal lulus
    4. Sistem generate hash SHA-256 dari data mahasiswa
    5. Sistem generate nomor ijazah: UNSUB/{prodi}/{tahun}/{random}
    6. Sistem generate QR Code berisi URL verifikasi
    7. Sistem simpan ijazah dengan status 'draft'
    8. Sistem update mahasiswa: ipk, tahun_lulus, status='lulus'
    9. Tampilkan notifikasi sukses
  Post-condition: Ijazah baru dengan status 'draft' tersimpan di database
  Skenario Alternatif:
    4a. Mahasiswa sudah punya ijazah -> tolak dengan error


  --- UC-03: Publish ke Blockchain ---
  Aktor: Admin Akademik
  Pre-condition: Ijazah berstatus 'draft'
  Trigger: Admin klik tombol "Publish" pada ijazah
  Skenario Normal:
    1. Sistem panggil BlockchainService::issueCertificate()
    2. (Mock Mode) Sistem generate tx hash random
    3. (Production) Sistem kirim transaksi ke smart contract Ethereum
    4. Catat transaksi di tabel blockchain_transactions
    5. Update ijazah: tx_hash, block, timestamp, status='issued'
    6. Generate PDF ijazah via DomPDF (include QR code & link Etherscan)
    7. Simpan file PDF di storage
    8. Hitung pdf_hash
    9. Tampilkan notifikasi sukses
  Post-condition: Ijazah terbit, hash tersimpan di blockchain, PDF tergenerate
  Skenario Alternatif:
    3a. Transaksi gagal -> catat log error, status transaksi 'failed'


  --- UC-04: Verifikasi via Hash ---
  Aktor: Visitor (Publik)
  Pre-condition: Tidak perlu login
  Trigger: Visitor memasukkan hash SHA-256 (64 karakter hex)
  Skenario Normal:
    1. Visitor memasukkan hash di form
    2. Sistem validasi format hash (harus 64 karakter hex)
    3. Sistem cari di tabel ijazah berdasarkan hash_sha256
    4. Sistem verifikasi via BlockchainService::verifyCertificate()
    5. Jika valid:
       - Tampilkan data mahasiswa (nama, NIM, prodi, IPK, fakultas)
       - Tampilkan status "VALID"
    6. Catat log verifikasi (IP, user agent, timestamp, hasil)
  Post-condition: Hasil verifikasi ditampilkan, log tersimpan
  Skenario Alternatif:
    3a. Hash tidak ditemukan -> tampilkan "Ijazah tidak ditemukan"
    4a. Hash revoked -> tampilkan "Ijazah telah dicabut"


  --- UC-05: Verifikasi via Upload File ---
  Aktor: Visitor (Publik)
  Pre-condition: Tidak perlu login
  Trigger: Visitor upload file PDF ijazah
  Skenario Normal:
    1. Visitor memilih file PDF (max 5MB)
    2. Sistem validasi file (tipe PDF, ukuran)
    3. Sistem hitung SHA-256 hash dari file yang diupload
    4. Sistem cari di tabel ijazah berdasarkan pdf_hash
    5. Sistem verifikasi via blockchain
    6. Tampilkan hasil (sama seperti verifikasi hash)
    7. Catat log verifikasi
  Post-condition: Hasil verifikasi ditampilkan
  Skenario Alternatif:
    2a. File bukan PDF -> tolak
    2b. File > 5MB -> tolak


  --- UC-06: Revoke Ijazah ---
  Aktor: Admin Akademik
  Pre-condition: Ijazah berstatus 'issued'
  Trigger: Admin klik "Revoke Certificate"
  Skenario Normal:
    1. Admin memilih ijazah yang akan di-revoke
    2. Sistem konfirmasi "Yakin ingin revoke?"
    3. Admin konfirmasi
    4. Sistem update status ijazah menjadi 'revoked'
    5. Sistem catat notes revokasi
    6. Tampilkan notifikasi sukses
  Post-condition: Ijazah status 'revoked', verifikasi akan menolak

11. PERANCANGAN ARSITEKTUR SISTEM
========================================================================

----------------------------------------------------------------------
11.1. Pola Arsitektur
----------------------------------------------------------------------
  Sistem menggunakan arsitektur berbasis layer (Layered Architecture)
  dengan pembagian:

  +========================================================================+
  |  LAYER 1: PRESENTATION LAYER (Frontend - Next.js)                     |
  |  - UI Components                                                       |
  |  - Client-side state management                                        |
  |  - Routing (App Router)                                                |
  |  - HTTP communication via Axios                                        |
  +========================================================================+
                |                 REST API (JSON)
                v
  +========================================================================+
  |  LAYER 2: APPLICATION LAYER (Backend - Laravel Controllers)           |
  |  - Request handling                                                    |
  |  - Validation                                                          |
  |  - Authentication & Authorization                                      |
  |  - Response formatting                                                 |
  +========================================================================+
                |
                v
  +========================================================================+
  |  LAYER 3: SERVICE LAYER (Laravel Services)                            |
  |  - Business logic (CertificateService)                                 |
  |  - Blockchain interaction (BlockchainService)                          |
  |  - PDF generation (DomPDF)                                             |
  |  - QR Code generation (Endroid QR Code)                                |
  +========================================================================+
                |
                v
  +========================================================================+
  |  LAYER 4: PERSISTENCE LAYER (Laravel Models + MySQL)                 |
  |  - Eloquent ORM                                                        |
  |  - MySQL database                                                      |
  |  - File storage (local/public)                                         |
  +========================================================================+

----------------------------------------------------------------------
11.2. Pola Desain (Design Patterns)
----------------------------------------------------------------------
  - MVC Pattern: Laravel (Model-View-Controller)
  - Service Layer Pattern: CertificateService, BlockchainService
  - Repository Pattern: via Eloquent ORM
  - Dependency Injection: Laravel Service Container
  - Singleton: BlockchainService (single instance)
  - DTO (Data Transfer Object): JSON responses dari API

----------------------------------------------------------------------
11.3. Strategi Komunikasi
----------------------------------------------------------------------
  Frontend <-> Backend:
    - RESTful API (JSON)
    - HTTP Methods: GET, POST, PUT, DELETE
    - Authentication: Bearer Token (Sanctum)
    - CORS: Sanctum stateful domains

  Backend <-> Blockchain:
    - JSON-RPC via web3p/web3.php
    - HTTP POST ke RPC URL (localhost:8545 untuk development)

  Backend <-> Database:
    - MySQL PDO via Laravel Eloquent ORM

  Backend <-> File Storage:
    - Laravel Filesystem (local/public disk)

----------------------------------------------------------------------
11.4. Strategi Keamanan
----------------------------------------------------------------------
  - Password: bcrypt (Laravel default)
  - API Auth: Sanctum token-based
  - CORS: terbatas ke frontend URL
  - Input Validation: Form Request / Validator
  - SQL Injection: Eloquent ORM (parameter binding)
  - XSS: Blade escaping ({{ }})
  - Blockchain: data immutable, hash verification
  - Rate Limiting: bisa ditambahkan di Laravel
